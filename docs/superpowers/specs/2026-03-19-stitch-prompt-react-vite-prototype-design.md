# Stitch Prompt React/Vite 高保真原型设计

## 概述

本次工作目标是在 `React + Vite` 中将 `design/stitch_prompt` 下的三类设计稿高保真落地为可操作前端原型：

- 登录页
- 工作台诊断页
- 工作台对比页

本次只实现一套主视觉，并通过主题切换覆盖 `light / dark` 两种模式。不单独实现 `bright mint`、`neon mint` 等额外视觉变体。

交付目标不是静态截图还原，而是“接近真实产品的前端原型”：支持拖拽分栏、版本切换、关闭标签、采纳、复制、对比、主题切换，以及本地 mock 数据驱动的页面流转。

## 范围

### 包含内容

- 使用 `React + Vite` 初始化单页应用
- 建立三类页面对应的路由和页面状态
- 实现单一主视觉 + `light/dark` 主题切换
- 实现本地 mock 登录流程
- 实现工作台输入、诊断、优化、版本管理、对比、复制、采纳等高保真交互
- 实现左右分栏拖拽与比例记忆
- 实现 toast、菜单、hover、focus、disabled、loading 等状态细节
- 保证桌面端大屏场景下的稳定展示与交互

### 不包含内容

- 设计目录中的多套色板变体逐一落地

## 技术方案

### 方案对比

#### 方案 A：React + Vite + 路由 + 本地状态管理（推荐）

优点：

- 最适合高保真还原和细节交互
- 代码结构清晰，后续接后端成本低
- 不受现成 UI 库默认样式束缚

缺点：

- 基础组件和状态需要自行组织

#### 方案 B：React + Vite + UI 组件库

优点：

- 初期搭建快

缺点：

- 容易偏离设计稿节奏与气质
- 组件库皮肤会影响 tab、按钮、输入区、diff 高亮等关键细节

#### 方案 C：静态页面拼装进 Vite

优点：

- 截图级静态还原最快

缺点：

- 不适合拖拽分栏、版本管理、对比模式等复杂交互
- 后续演进成本高

### 最终选择

采用 **方案 A**：在 `React + Vite` 中构建一套自定义但不过度抽象的组件层，兼顾 1:1 还原、交互质量和后续可维护性。

## 信息架构

### 路由

- `/login`
- `/workbench/diagnostics`
- `/workbench/comparison`

说明：

- 登录成功默认跳转到 `/workbench/diagnostics`
- 对比页作为工作台的独立高保真展示入口，同时共享工作台状态与组件

### 页面关系

- 登录页负责认证入口和品牌展示
- 工作台诊断页展示默认编辑 + 诊断结果布局
- 工作台对比页展示当前 prompt 与版本对比布局

三个页面共用主题系统、品牌元素、核心按钮风格、字体体系和部分基础组件。

## 组件设计

### 全局层

#### `ThemeProvider`

- 管理 `light/dark` 主题
- 从 `localStorage` 读取和写入用户偏好
- 通过 `data-theme` 和 CSS variables 驱动全局样式

#### `AppShell`

- 负责应用级布局、路由出口和全局 UI 容器
- 为工作台页挂载 toast 区域、下拉菜单容器等

#### `ToastHost`

- 显示复制成功、超出版本上限、模拟错误等轻提示

### 登录页

#### `LoginPage`

- 品牌图标、标题、副标题
- 登录表单卡片
- 用户名和密码输入框
- 登录按钮
- 注册/忘记密码辅助链接
- 页脚链接和版权信息

### 工作台公共层

#### `TopBar`

- 左侧品牌
- 模型选择器
- 主题切换按钮
- 用户头像和退出菜单

#### `StatusBar`

- 底部状态信息
- 在对比页还原 “Ready / System Online” 一类状态展示

#### `WorkbenchLayout`

- 顶栏 + 主内容区 + 底栏
- 内含左右分栏与右侧内容切换

#### `SplitPane`

- 默认比例约 `40 / 60`
- 支持鼠标拖拽调整
- 限制左右最小宽度，避免布局塌陷
- 拖拽结束后记忆比例

### 左侧输入区

#### `PromptEditorPanel`

- 区域标题
- 字符统计
- 多行编辑器
- `Diagnose` / `Optimize` 操作区
- 支持 `Tab` 缩进
- 支持空白输入禁用按钮
- 请求进行中展示 loading 和并发禁用

### 右侧结果区

#### `ResultTabs`

- `Diagnostics` 固定标签
- `Version N` 动态标签
- 关闭标签按钮
- 激活态下划线与 hover 状态

#### `DiagnosticsPanel`

- 诊断卡片列表
- 卡片包含问题类型、说明、建议
- 底部主按钮：`Optimize based on diagnostics`

#### `VersionPanel`

- 展示优化后 prompt 文本
- 底部操作：`Adopt`、`Copy`、`Compare`

#### `DiffPanel`

- 内联词级 diff 展示
- 红色表示删除，绿色表示新增
- 顶部 `Exit Comparison`

## 状态设计

### 应用级状态

- 当前主题
- 当前用户会话
- 当前选中模型
- 全局 toast

### 工作台状态

- 左侧 prompt 文本
- 字符数
- 当前分栏比例
- 诊断结果数组
- 版本列表
- 当前激活 tab
- 当前是否处于对比模式
- 当前对比的版本 ID
- 当前请求状态：空闲 / 诊断中 / 优化中

### 瞬时 UI 状态

- 头像菜单开关
- 按钮复制成功反馈
- hover/focus/pressed 表现

## 交互流程

### 登录流程

1. 用户访问 `/login`
2. 输入用户名和密码
3. 本地校验非空
4. 点击登录后显示 loading
5. mock 登录成功后写入本地会话并跳转 `/workbench/diagnostics`
6. 点击头像菜单中的退出后清空会话并返回 `/login`

### 诊断流程

1. 用户在左侧输入 prompt
2. 点击 `Diagnose`
3. 进入模拟请求中状态
4. 右侧创建或刷新 `Diagnostics` 标签内容
5. 已有版本标签保持不变

### 优化流程

1. 用户点击 `Optimize`
2. 进入模拟请求中状态
3. 新增一个 `Version N` 标签
4. 自动切换到最新版本
5. 当版本数达到 10 个时，阻止继续新增并 toast 提示

### 基于诊断优化

1. 用户在 `Diagnostics` 面板点击 `Optimize based on diagnostics`
2. 使用当前诊断结果参与 mock 版本生成
3. 新增版本标签并切换到该版本

### 版本操作

#### `Adopt`

- 将当前版本内容回填到左侧编辑器
- 同步刷新字符数

#### `Copy`

- 复制版本文本到剪贴板
- 显示成功 toast 或按钮成功反馈

#### `Compare`

- 进入对比模式
- 右侧切换为 `DiffPanel`
- 顶部出现 `Exit Comparison`

#### 关闭标签

- 支持关闭任意版本标签
- 若关闭的是当前激活标签，则切换到相邻可用标签

### 分栏拖拽

- 鼠标按下分割线开始拖拽
- 持续更新左右宽度
- 设置安全边界，避免任一栏过窄
- 在两个工作台页面间共享上次调整结果

## 视觉系统

### 主题策略

仅实现一套主视觉结构，通过 token 切换 `light/dark`。

### 颜色令牌

亮色：

- 页面背景：`#F8FFFE`
- 面板背景：`#FFFFFF`
- 主文字：`#1E293B`
- 次文字：`#64748B`
- 强调色：`#0D9488`
- 辅助强调：`#E11D48`
- 边框：`#E2E8F0`

暗色：

- 页面背景：`#0F172A`
- 面板背景：`#1E293B`
- 主文字：`#F1F5F9`
- 次文字：`#94A3B8`
- 强调色：`#2DD4BF`
- 辅助强调：`#FB7185`
- 边框：`#334155`

### 字体

- UI 字体：优先选择接近设计稿观感的无衬线字体
- 编辑器字体：等宽字体
- 如运行环境不可用，则降级到系统字体栈

### 视觉约束

- 登录卡片、工作台卡片、诊断卡片共享统一圆角节奏
- 保持设计稿中的轻量阴影和低对比边框
- 按钮层级明确：主操作为 mint，关键操作 `Adopt` 为 rose
- tab 活跃态使用薄荷色强调线
- diff 高亮在 light/dark 下分别校准透明度与文字对比度

## Mock 数据策略

### 登录

- 使用固定成功 mock
- 可保留一个失败分支用于错误提示验证

### 模型列表

- 本地提供 2~3 个模型项
- 包含可用和不可用状态

### 诊断结果

- 以设计稿中的 3 张问题卡片为基础准备默认 mock 数据

### 版本数据

- 准备至少两个优化版本，便于还原 `Version 1 / Version 2`
- 对比页默认展示一个已生成版本及对应 diff

## 验收标准

- 三个页面均可独立访问
- `light/dark` 主题切换生效且持久化
- 登录、退出、模型切换、诊断、优化、版本切换、关闭 tab、复制、采纳、对比、退出对比、拖拽分栏全部可操作
- 页面在设计稿目标桌面尺寸下视觉结构与设计保持一致
- 较窄桌面宽度下不出现明显破版
- `vite build` 可通过

## 实现边界

- 不接真实 API，但结构上为未来替换 mock 留出清晰边界
- 不实现真实 SSE，但保留可替换的异步调用入口
- 不在本次范围中加入历史页、项目页、移动端深度适配

## 风险与对策

### 风险 1：没有现成前端工程

对策：直接初始化最小可用 `React + Vite` 工程，控制依赖数量，避免为原型引入过重技术栈。

### 风险 2：高保真还原容易被通用组件风格污染

对策：不引入大型 UI 组件库，核心组件自行实现。

### 风险 3：对比视图细节容易偏离设计稿

对策：以对比页截图为主基准，优先还原文字区留白、按钮位置、顶部层级和底部状态栏。
